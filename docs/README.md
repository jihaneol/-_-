# Project Brain

`docs/` is the planning source of truth. A feature must be shaped here before it becomes an executable phase.

## What

`docs/what/` answers what this project is building.

- Goal and target role proof.
- MVP scope and explicitly excluded work.
- Prioritized roadmap.
- Operator UI goals and screen map.

## How

`docs/how/` answers how the project is implemented.

- Data flow and architecture.
- Domain model and API contract.
- Backend and frontend patterns.
- Test strategy and code references.

## Why

`docs/why/` answers why the implementation is shaped this way.

- Capability mapping.
- Tradeoffs and decision records.
- Review loop and development evidence.

## Harness

`docs/harness/` and `docs/frontend-harness/` are portfolio harness summaries. They do not replace `what/how/why`; they collect target-role proof, scope, architecture, test evidence, review loops, and week plans in a format that is easy to audit.

## Operations

`docs/operations/` contains inboxes, candidate lists, checklists, and week plans. These files help run the work, but they are not implementation contracts by themselves.
